# 자동 검수 아키텍처

## 책임 경계

| 계층 | 책임 |
| --- | --- |
| 형식별 파서 | 원본 파일을 표·셀·메타정보의 중립 모델로 변환 |
| 가져오기 저장소 | 연보와 모든 표·셀을 한 트랜잭션으로 저장 |
| 프로파일 저장소 | 구조 서명 비교, 기존 프로파일 승계, 새 프로파일 저장 |
| 계산 검수 워크플로 | 프로파일 준비 후 결정론적 규칙 엔진 실행 |
| 언어 검수 워크플로 | 파란색 후보, 오탈자·번역 후보, 사전·캐시·LLM 처리 |
| 처리 작업 워크플로 | import → calculation → language 단계를 기록·재시도 |
| 조회/표시 계층 | 검수 결과를 사용자용 그룹과 하이라이트 좌표로 변환 |

이 경계 덕분에 파서, 계산 규칙, LLM 모델, 화면 표시를 서로 독립적으로 교체할 수
있습니다. 검수 엔진은 원본 파일을 읽지 않고 DB의 중립 모델과 프로파일만 읽습니다.

### 구현 모듈

| 모듈 | 역할 |
| --- | --- |
| `validation/profiles.py` | 공유·템플릿·표별 생성기를 순서대로 실행해 프로파일 초안 생성 |
| `validation/curated_profiles.py` | 확인된 특수 표의 명시 규칙 병합 |
| `validation/catalog.py` | 검수 종류, 중요도, 저장 가능한 명세 종류, 명세별 실행기 관리 |
| `validation/profile_repository.py` | 구조 서명 비교와 프로파일 저장·승계 |
| `validation/profile_rules.py` | DB에서 읽은 명세를 카탈로그의 실행기에 전달 |
| `validation/engine.py` | 표 단위 규칙과 교차표 규칙을 실행하고 결과 수집 |
| `validation/spec_repository.py` | 검수 결과의 `rule_id`를 저장된 프로파일 명세와 다시 연결 |
| `validation/highlights.py` | 명세와 검수 좌표로 목표·연산 입력 셀 계산 |
| `validation/presentation.py` | 내부 검수 결과를 API 표시 모델로 변환 |

## 1. 파일 입력과 DB 구성

`ReportProcessingWorkflow`가 업로드 작업을 만들고 원본 확장자에 맞는 파서를
선택합니다.

- Excel: 병합 셀 값을 펼치고, 표 번호가 반복되는 지점으로 한 시트의 여러 표를
  분리합니다. `표번호`, `통계명`, `단위`, `기준일`, `주석`, `출처`, `분야` 행은
  메타정보로 인식합니다.
- HWPX: 문서 XML, 병합 셀, 각주와 원문 스타일 정보를 이용합니다.
- Markdown: 변환된 표와 주변 문단의 메타정보를 연결합니다.

파서 결과는 `ImportedTable` 형식으로 정규화합니다. Excel은
`ReportImportRepository`가 `annual_reports`, `stat_tables`, `stat_table_cells`를
한 트랜잭션으로 저장합니다. 중간 실패 시 불완전한 연보가 남지 않습니다. HWPX와
Markdown도 같은 DB 계약을 사용하므로 이후 단계에는 파일 형식 차이가 노출되지
않습니다.

## 2. 구조 변경 감지

각 표의 구조 서명에는 다음 정보가 들어갑니다.

- 헤더 행 수와 전체 행·열 수
- 병합을 해석한 헤더 텍스트
- 라벨 열 위치
- 데이터 행 라벨의 순서

값만 다음 연도로 바뀌면 구조 서명은 유지됩니다. 행·열, 헤더 또는 항목 구성이
바뀌면 새 서명이 만들어집니다. 표 번호만 이동한 경우에는 동일 제목과 구조
서명으로 기존 프로파일을 찾아 자동 승계합니다.

## 3. 검수 프로파일 생성

프로파일은 실행 가능한 검수 명세이며 DB의 `validation_profiles.rules_json`에
저장됩니다. 생성 순서는 다음과 같습니다.

1. **공유 규칙**: 메타정보 존재 여부처럼 모든 표에 적용할 규칙을 추가합니다.
2. **표 유형 템플릿**: 지역별 표처럼 구조적 특징이 확인된 표에 템플릿 규칙을
   추가합니다.
3. **통계표별 추론**: 행·열 라벨, 헤더 계층, 실제 값의 일치 비율을 이용해 합계,
   소계, 산식, 비율, 증감률, 평균 후보를 만듭니다.
4. **큐레이션 규칙**: 이미 확인한 특수 표의 명시 규칙을 덮어씌웁니다.
5. **선택적 LLM 보강**: 새 표 또는 구조 변경 표만 LLM이 산식 명세를 재검토합니다.
6. **저장 및 재사용**: 같은 표 번호와 구조 서명이 다시 들어오면 저장된 프로파일을
   그대로 사용합니다.

프로파일 생성이 한 표에서 실패해도 전체 연보 처리는 중단하지 않습니다. 해당
표에는 공유 규칙과 이상치 규칙만 적용한 `needs_review` 폴백 프로파일을 저장하고,
실패 원인을 프로파일 메모에 남깁니다.

저장 가능한 명세 종류와 실제 실행 메서드의 연결은 `validation/catalog.py`의
`PROFILE_SPEC_EXECUTORS`가 단일 기준입니다. 새 산식 종류를 추가할 때 이 카탈로그,
실행 메서드와 하이라이트 전략을 함께 추가하며, 테스트가 존재하지 않는 실행기
이름이나 LLM 전용 명세 누락을 검사합니다.

### 공유 프로파일과 통계별 프로파일

공유 규칙은 코드에 등록된 생성기 목록으로 관리합니다. 표 유형 템플릿은 감지된
유형에 해당하는 생성기만 실행합니다. 통계별 프로파일은 이 결과와 표 자체에서
추론한 산식 및 큐레이션 산식을 합친 최종 결과입니다. 실제 검수 시 Python의 추론
함수를 다시 실행하는 것이 아니라 DB에 저장된 최종 명세를 읽어 실행합니다.

## 4. 합계·비율 검수

`CalculationValidationWorkflow`는 반드시 요청받은 `report_id`의 표와 프로파일만
불러옵니다. 최신 연보를 묵시적으로 다시 찾지 않으므로 업로드가 겹쳐도 다른
연보의 표가 섞이지 않습니다.

규칙 엔진은 다음 계열을 처리합니다.

- 행 방향 합계와 열 방향 합계
- 총계 → 직속 소계 → 세부 항목의 계층 합계
- 고정 셀 조합과 헤더에 표시된 사칙연산
- 분자/분모 비율, 구성비, 총계의 재계산 비율
- 전년 대비 증감액·증감률
- 행 기반 연도 추이와 가중 평균
- 분리된 하위표 사이의 교차표 합계·일치 검수

빈값과 `-`도 프로파일의 연산 대상 좌표에는 남습니다. 현재 숫자가 없으면 계산을
억지로 통과시키지 않지만, 다음 연도에 값이 들어올 위치로 계속 추적합니다.

## 5. 오탈자·번역과 파란색 표기

언어 단계는 계산 단계와 분리되어 있어 외부 API 오류 시 계산 결과를 잃지 않고
언어 단계만 재시도할 수 있습니다.

1. HWPX의 파란색 원문 위치를 DB 셀 또는 제목에 연결합니다.
2. 제목·헤더·텍스트 셀에서 오탈자와 번역 후보를 만듭니다.
3. 메타정보 본문과 숫자뿐인 파란색 값은 언어 후보에서 제외합니다.
4. 공식 기관명·행정구역명·표준어 사전의 정확 일치를 먼저 적용합니다.
5. 표·행·열 문맥, 프롬프트 버전, 모델을 포함한 캐시 키로 이전 결과를 재사용합니다.
6. 남은 후보만 한 문맥당 한 요청으로 묶어 LLM에 보냅니다.
7. 응답 ID와 원본 셀 값을 다시 대조한 뒤에만 결과를 저장합니다.

LLM 응답 일부가 빠지거나 원본과 연결되지 않으면 임의의 셀에 저장하지 않고 해당
후보를 대기 상태로 남깁니다. 전수 LLM 모드에서는 대기 후보가 하나라도 있으면
언어 단계를 실패 처리해 부분 결과를 완료로 오인하지 않습니다.

## 6. 하이라이트

규칙 실행 결과에는 목표 셀과 연산 입력 셀의 좌표가 함께 기록됩니다. 조회 계층은
규칙 종류별로 다음 역할을 부여합니다.

- `target`: 합계·비율 등 산출 결과 셀. 강한 하이라이트를 사용합니다.
- `related`: 해당 결과를 계산하는 데 실제로 사용한 가장 가까운 단계의 셀. 약한
  하이라이트를 사용합니다.

좌표 계산은 DB 조회 코드와 분리된 `validation/highlights.py`의 순수 함수가
담당합니다. 따라서 SQLite와 PostgreSQL 중 어느 DB를 사용하더라도 같은 프로파일
명세와 검수 행에는 같은 하이라이트 좌표가 생성됩니다. 규칙 명세 조회는
`validation/spec_repository.py`, API 모델 변환은 `validation/presentation.py`가
각각 담당합니다.

표시 그룹은 같은 산식을 반복 적용한 정상 검수를 하나로 합치되, 포함된 모든 셀
좌표는 유지합니다. 합계 검수 안에서는 상위 총계를 먼저, 그다음 소계를 보여주고,
행 방향·열 방향처럼 같은 방향의 계산을 연속 배치합니다. 문제 검수도 같은 산식과
상태면 하나로 묶되 문제 목표 셀은 각각 추적할 수 있습니다.

## 7. 실패와 재시도

각 단계의 시도 횟수, 결과와 오류는 `report_processing_stages`에 별도로 저장됩니다.

- import 실패: 파일 입력부터 다시 실행
- calculation 실패: 이미 저장한 연보를 재사용하고 계산부터 다시 실행
- language 실패: 저장된 연보와 계산 run을 재사용하고 언어 단계만 다시 실행

LLM 전송 계층은 408, 409, 429와 일시적인 5xx 오류에 지수 백오프로 재시도합니다.
구조 해석 또는 LLM 프로파일 보강 실패는 한 표만 폴백 처리합니다. 파일 자체가
깨졌거나 지원하지 않는 구조처럼 복구할 수 없는 오류는 작업 상태 API에 단계와
원인을 남깁니다.

PoC는 FastAPI 프로세스의 BackgroundTasks를 사용합니다. 여러 서버 인스턴스와
프로세스 재시작까지 자동 복구해야 하는 운영 환경에서는 같은
`ReportProcessingWorkflow`를 Celery, RQ 또는 별도 작업 큐 worker에서 호출하면
됩니다. DB 단계 기록과 재시도 경계는 그대로 재사용할 수 있습니다.
